## 目的

- 外部APIを用いた非同期通信の実装

- インターフェースによる実装の抽象化と切り替え構造の習得
- Vitestを用いたテスト設計（スタブ含む）
- 現場でも活用できる保守性・拡張性のある設計パターンの体得

## 基本要件（Fake Store API × localStorage）

### 商品一覧ページ

- Fake Store API から商品一覧を取得し、カード形式で表示する
- 商品には「詳細を見る」ボタンを設置し、詳細ページへ遷移できるようにする
- 通信中はローディング表示、失敗時はエラーメッセージを表示する
- データ取得処理は必ずインターフェース経由で行うこと

### 商品詳細ページ

- 商品IDを元に外部APIから商品情報を取得して表示する
- 「カートに追加」ボタンで Pinia ストアへ追加する
- 「購入」ボタンで Pinia の購入履歴ストアに追加し、localStorage に保存する
- 商品取得・保存処理はインターフェース経由で行うこと

### カートページ

- カートに追加された商品を一覧表示する
- 各商品に「削除」ボタンを設置し、カートから除外できるようにする
- カート内商品の合計金額を表示する
- カートの状態は Pinia に一時的に保持し、localStorage には保存しない

### 購入履歴ページ

- アプリ起動時、localStorage から購入履歴を取得して Pinia に反映する
- 一覧で購入済み商品を表示する
- 各商品に「削除」ボタンを設置し、削除後は localStorage を更新する
- 購入履歴の取得・追加・削除処理は、インターフェース経由で行うこと

### インターフェース設計（必須）

- 商品取得および購入履歴操作に関して、すべてインターフェースを定義し、それに準拠した実装を行うこと
- 利用側（画面・ストア）は、実装詳細に依存しないよう設計すること（抽象化）

## 発展課題（スタブAPI実装とスタブテスト）

### 目的

- スタブAPIの作成・活用により、非同期処理への理解を深める
- スタブによるテスト制御の重要性と実用性を体験する
- 実装とテストの責務を明確にし、切り替え可能な柔軟設計を学ぶ

---

### スタブAPIの実装
- 商品取得処理用のスタブを用意する
    - 商品データは仮のものを用意する
    - 通信失敗や遅延など、さまざまなケースを想定できるような柔軟な実装とする
- 画面上にボタンを用意し、ボタン押下でFake Store APIからのデータ取得とスタブからのデータ取得を切り替えられるようにする

### スタブを使ったテストの実施

- テスト時にはスタブを使用する
- 非同期処理やエラー発生時の動作を意図的に再現し、正しく処理できるか確認する
- APIの依存を排除したテストにより、テストの信頼性と保守性を実感する
